## 1. System Overview

### 1.1 Purpose

CALM Hub is a centralized REST API backend for storing, versioning, and retrieving **Common Architecture Language Model (CALM)** artifacts. It serves as the authoritative registry for architecture definitions, design patterns, decision records, compliance controls, and related metadata within an organization — particularly targeting regulated environments such as financial services.

### 1.2 Key Capabilities

| Capability | Description |
|------------|-------------|
| **Multi-Tenant Namespace Management** | Logical isolation of resources via namespaces (e.g., `finos`, `traderx`) with dot-separated hierarchical naming |
| **Versioned Artifact Storage** | Semantic versioning (MAJOR.MINOR.PATCH) for architectures, patterns, flows, standards, interfaces, controls, and schemas |
| **Architecture Decision Records** | Revision-based ADR lifecycle management with status transitions (draft → proposed → accepted → deprecated/superseded/rejected) |
| **Compliance Control Framework** | Hierarchical domain → control → requirement → configuration structure for governance artifacts |
| **Decorator / Metadata Augmentation** | Ability to attach typed metadata decorators to resources with filtering by target and type |
| **Human-Readable Resource Addressing** | Front Controller pattern providing kebab-case slug URLs (e.g., `/calm/namespaces/finos/my-pattern`) alongside legacy numeric IDs |
| **Dual Storage Backends** | Pluggable persistence via MongoDB (production) or NitriteDB (standalone/embedded), selected at runtime |
| **OAuth2/OIDC Security** | Optional Keycloak-backed JWT authentication with scope-based authorization and namespace-level RBAC |
| **CALM Schema Registry** | Publication and retrieval of versioned CALM JSON Schema definitions |
| **OpenAPI Documentation** | Auto-generated Swagger UI available at runtime |

### 1.3 Primary Users / Actors

| Actor | Description |
|-------|-------------|
| **Architect** | Creates and manages architecture definitions, patterns, flows, standards, interfaces, and ADRs within namespaces |
| **Developer** | Retrieves architecture artifacts and schemas to understand system design; may publish artifacts via CI/CD |
| **Compliance Officer** | Manages control domains, requirements, and configurations for governance enforcement |
| **Platform Administrator** | Creates namespaces, manages user access grants, and configures deployment (storage mode, security profile) |
| **CI/CD Pipeline** | Automated actor that publishes architecture artifacts and schemas via the REST API using machine-to-machine OAuth tokens |
| **Security Administrator** | Configures OAuth scopes, manages Keycloak realm, and assigns RBAC grants to users |

## 4. Functional Requirements

### 4.1 REST API Structure

The API is rooted at `/calm` and organized into the following endpoint groups:

| Endpoint Group | Base Path | Description |
|----------------|-----------|-------------|
| Namespaces | `/calm/namespaces` | Namespace CRUD |
| Architectures | `/calm/namespaces/{namespace}/architectures` | Architecture CRUD + versioning |
| Patterns | `/calm/namespaces/{namespace}/patterns` | Pattern CRUD + versioning |
| Flows | `/calm/namespaces/{namespace}/flows` | Flow CRUD + versioning |
| Standards | `/calm/namespaces/{namespace}/standards` | Standard CRUD + versioning |
| Interfaces | `/calm/namespaces/{namespace}/interfaces` | Interface CRUD + versioning |
| ADRs | `/calm/namespaces/{namespace}/adrs` | ADR CRUD + revision tracking |
| Decorators | `/calm/namespaces/{namespace}/decorators` | Decorator CRUD + filtering |
| Schemas | `/calm/schemas` | Schema publication + retrieval |
| Domains | `/calm/domains` | Control domain management |
| Controls | `/calm/domains/{domain}/controls` | Control requirement + configuration management |
| Front Controller | `/calm/namespaces/{namespace}/{customId}` | Slug-based resource access |
| User Access | `/calm/namespaces/{namespace}/user-access` | RBAC grant management |
| Mappings | `/calm/namespaces/{namespace}/mappings` | Custom ID ↔ numeric ID lookup |

### 4.2 Versioned Resource Lifecycle (Architectures, Patterns, Flows, Standards, Interfaces)

These five resource types share an identical lifecycle:

1. **Create**: `POST /calm/namespaces/{namespace}/{resourceType}` — Creates the resource at version `1.0.0`. Returns `201 Created` with a `Location` header.
2. **List**: `GET /calm/namespaces/{namespace}/{resourceType}` — Returns all resources in the namespace as summary objects.
3. **List Versions**: `GET /.../{resourceId}/versions` — Returns all version strings for a resource.
4. **Get Version**: `GET /.../{resourceId}/versions/{version}` — Returns the JSON content at a specific version.
5. **Create Version**: `POST /.../{resourceId}/versions/{version}` — Creates a new version; returns `409 Conflict` if the version already exists.
6. **Update Version**: `PUT /.../{resourceId}/versions/{version}` — Updates content in-place; returns `403 Forbidden` if `allow.put.operations` is `false`.

**Version format**: Semantic versioning `MAJOR.MINOR.PATCH` (e.g., `1.0.0`, `2.3.1`).  
**Internal storage**: Versions are stored with hyphens (`1-0-0`) in MongoDB and converted to dots (`1.0.0`) for the API.

### 4.3 ADR Lifecycle

ADRs differ from versioned resources in that they use **incremental revisions** instead of semantic versions:

1. **Create**: `POST /calm/namespaces/{namespace}/adrs` — Creates ADR at revision 1.
2. **Update**: `POST /calm/namespaces/{namespace}/adrs/{adrId}` — Creates a new revision (revision N+1) with updated content.
3. **Change Status**: `POST /calm/namespaces/{namespace}/adrs/{adrId}/status/{status}` — Changes the ADR status and creates a new revision.
4. **Get Latest**: `GET /.../{adrId}` — Returns the latest revision.
5. **List Revisions**: `GET /.../{adrId}/revisions` — Returns all revision numbers.
6. **Get Revision**: `GET /.../{adrId}/revisions/{revision}` — Returns a specific revision.

**ADR Status Values**: `draft`, `proposed`, `accepted`, `deprecated`, `superseded`, `rejected`.

**ADR Content Fields**: title, contextAndProblemStatement, decisionDrivers (array of strings), consideredOptions (array of Option objects with name/description/pros/cons), decisionOutcome (chosenOption + rationale), links (array of Link objects with rel/href).

**HTML Sanitization**: All ADR text fields are sanitized using OWASP HTML policy (formatting + blocks + tables allowed; scripts and other dangerous content removed).

### 4.4 Control Hierarchy

Controls follow a three-level hierarchy:

```
Domain (e.g., "security")
  └── Control (has a requirement)
       ├── Requirement Versions (1.0.0, 1.1.0, ...)
       └── Configurations (multiple per control)
            └── Configuration Versions (1.0.0, 1.1.0, ...)
```

1. **Create Domain**: `POST /calm/domains` — Creates a named domain.
2. **Create Control**: `POST /calm/domains/{domain}/controls` — Creates a control with its first requirement (version 1.0.0).
3. **Version Requirement**: `POST /.../{controlId}/requirement/versions/{version}` — Creates new requirement version.
4. **Create Configuration**: `POST /.../{controlId}/configurations` — Creates a configuration at version 1.0.0.
5. **Version Configuration**: `POST /.../{controlId}/configurations/{configId}/versions/{version}` — Creates new configuration version.

### 4.5 Front Controller (Custom ID Routing)

The Front Controller provides a type-agnostic API layer using human-readable slugs:

**Custom ID Format**: `^[a-z][a-z0-9]*(-[a-z0-9]+)*$` (kebab-case, must start with a lowercase letter)

**Create Flow**:
1. Client POSTs to `/calm/namespaces/{namespace}/{customId}` with `type`, `json`, and optional `name`/`description`.
2. System creates a `ResourceMapping` (customId → numericId + resourceType).
3. System dispatches to the appropriate type-specific store (Pattern, Architecture, Flow, Standard, or Interface).
4. Resource is created at version `1.0.0`.

**Update Flow**:
1. Client POSTs to the same custom ID endpoint with `changeType` (`MAJOR`, `MINOR`, or `PATCH`) and `json`.
2. System looks up existing mapping to determine resource type and numeric ID.
3. System retrieves all existing versions, finds the latest using Semver comparison.
4. System bumps the version according to `changeType`:
   - `MAJOR`: N.0.0 → N+1.0.0
   - `MINOR`: N.M.0 → N.M+1.0
   - `PATCH`: N.M.P → N.M.P+1
5. System creates the new version in the type-specific store.

**$id Rewriting**: When `calm.hub.base-url` is configured, GET responses have their `$id` JSON field rewritten to a friendly URL:  
`{baseUrl}/calm/namespaces/{namespace}/{customId}/versions/{version}`

### 4.6 Input Validation

| Parameter | Regex Pattern | Example Valid | Example Invalid |
|-----------|--------------|---------------|-----------------|
| Namespace | `^[A-Za-z0-9-]+(\.[A-Za-z0-9-]+)*$` | `finos`, `my.namespace.test` | `invalid!!`, `_leading` |
| Domain Name | `^[A-Za-z0-9-]+$` | `security`, `data-governance` | `invalid!!` |

### 4.7 Error Response Conventions

| HTTP Status | Condition | Response Body Format |
|-------------|-----------|---------------------|
| **400** | Invalid input (bad JSON, regex mismatch, missing fields) | Plain text message |
| **403** | PUT operations disabled or insufficient authorization scope | Plain text message |

### 5.2 Architectural Patterns

#### 5.2.1 CDI Producer-Based Storage Backend Selection

Storage implementations are selected at runtime via CDI producers, controlled by `calm.database.mode`:

```
Store Interface (e.g., ArchitectureStore)
    ↓
StoreProducer (e.g., ArchitectureStoreProducer)
    ├── calm.database.mode = "mongo"      → MongoArchitectureStore
    └── calm.database.mode = "standalone" → NitriteArchitectureStore
```

13 store interfaces each have corresponding MongoDB and NitriteDB implementations, paired via 13 CDI producer classes.

#### 5.2.2 Layered Architecture

```
┌─────────────────────────────────────────┐
│           REST Resources (JAX-RS)       │  ← HTTP request handling, validation
├─────────────────────────────────────────┤
│           Store Interfaces              │  ← Business logic contract
├──────────────────┬──────────────────────┤
│  MongoDB Stores  │  NitriteDB Stores    │  ← Persistence implementations
├──────────────────┴──────────────────────┤
│           CDI Producers                 │  ← Runtime backend selection
├─────────────────────────────────────────┤
│         Security Filter (optional)      │  ← JWT + scope + RBAC enforcement
└─────────────────────────────────────────┘
```

#### 5.2.3 Document Model (MongoDB)

Resources are stored as namespace-scoped documents with nested arrays:

```json
{
  "namespace": "finos",
  "architectures": [
    {
      "architectureId": 1,
      "name": "My Architecture",
      "versions": {
        "1-0-0": { /* JSON content */ },
        "1-1-0": { /* JSON content */ }
      }
    }
  ]
}
```

Version keys use hyphens (`1-0-0`) because MongoDB does not allow dots in field names.

#### 5.2.4 Atomic ID Generation

A dedicated `counters` collection maintains per-resource-type sequences:

```json
{
  "patternStoreCounter": 1,
  "architectureStoreCounter": 1,
  "adrStoreCounter": 1,
  "standardStoreCounter": 1,
  "flowStoreCounter": 1,
  "controlStoreCounter": 2,
  "decoratorStoreCounter": 4,
  "interfaceStoreCounter": 2,
  "userAccessStoreCounter": 6,
  "resourceMappingStoreCounter": 1
}
```

IDs are generated atomically using MongoDB's `findOneAndUpdate` with `$inc` to ensure no duplicates under concurrency. NitriteDB uses in-memory `AtomicInteger` counters with `ReentrantLock` for thread safety.

#### 5.2.5 Version Conflict Prevention

New version creation uses MongoDB's atomic conditional updates:

```
db.architectures.updateOne(
  { "namespace": "finos", "architectures.architectureId": 1, "architectures.versions.1-0-0": { $exists: false } },
  { $set: { "architectures.$.versions.1-0-0": content } }
)
```

If the version field already exists, the `$exists: false` condition fails, and the store throws `VersionExistsException` (mapped to HTTP 409).

### 7.1 Performance

| Aspect | Approach | Evidence |
|--------|----------|----------|

## 8. Ambiguities and Gaps

### 8.1 Partially Implemented RBAC

**Observation**: The `AccessControlFilter` contains a TODO comment indicating that full RBAC checks are still pending implementation. The `UserAccessValidator` is functional but the `isUserAuthorized()` method has known limitations:

- `GET /calm/namespaces` is treated as a "default accessible" endpoint that bypasses RBAC for all authenticated users, with a comment noting this needs namespace-scoped filtering.
- The relationship between OAuth scopes and RBAC grants is not clearly documented — both layers are checked, but the interaction semantics (override vs. conjunction) are implicit.

**Impact**: In secure deployments, users with valid scopes may access resources in namespaces they should not have access to.

### 8.2 No DELETE Operations

**Observation**: No REST endpoint supports `DELETE` for any resource type. Once created, architectures, patterns, flows, standards, interfaces, ADRs, decorators, domains, namespaces, and schemas cannot be removed through the API.

**Impact**: Cleaning up test data, correcting mistakes, or decommissioning resources requires direct database manipulation.

### 8.3 PUT Operations Toggle

**Observation**: PUT (in-place update) is gated behind the `allow.put.operations` configuration property. When disabled (which appears to be the default), all PUT endpoints return `403 Forbidden`. The rationale for this toggle — whether it enforces immutability as a policy decision or is a safety mechanism — is not documented.

**Impact**: Operators must understand this configuration to enable or disable mutable semantics.

### 8.4 No Pagination

**Observation**: All list endpoints return the full set of results wrapped in `ValueWrapper`. There is no support for pagination (offset/limit, cursor-based, or page-based).

**Impact**: Performance degrades as namespace resource counts grow. Large namespaces with hundreds of architectures will produce large response payloads.

### 8.5 No Search or Query Capabilities

**Observation**: Aside from decorator filtering (by `target` and `type`), there is no full-text search, tag-based filtering, or query language for discovering resources by content or metadata.

**Impact**: Users must list all resources and filter client-side for discovery beyond basic namespace browsing.

### 8.6 Schema Versioning vs. Resource Versioning

**Observation**: Core schemas use their own versioning (e.g., `"2025.01"`) which does not follow the SemVer convention used by other resources. The relationship between schema versions and the resources they validate is not tracked in the data model.

**Impact**: There is no way to query which schema version an architecture or pattern was validated against.

### 8.7 Decorator Target/Type Opacity

**Observation**: Decorator filtering supports `target` and `type` query parameters, but the API does not define what valid values are. These appear to be free-form fields within the decorator JSON, and filtering is performed at the database layer.

**Impact**: Clients have no schema or enumeration for valid decorator targets and types; interoperability depends on convention.

### 8.8 Inconsistent Mutability

**Observation**: Some domain classes are immutable with Builder patterns (`Architecture`, `Pattern`, `Flow`, `Decorator`, `ResourceMapping`) while others are mutable POJOs (`Standard`, `CalmInterface`, `UserAccess`, `Domain`). The reason for this inconsistency is not documented.

**Impact**: Developers must check each entity individually to understand its threading and safety characteristics.

### 8.9 Front Controller and Legacy API Coexistence

**Observation**: Resources can be accessed both via the Front Controller (`/calm/namespaces/{namespace}/{customId}`) and via legacy numeric-ID endpoints (`/calm/namespaces/{namespace}/architectures/{id}`). Both paths write to the same underlying stores, but it is unclear whether resources created via one path are fully accessible via the other, or whether the mapping is bidirectional.

**Impact**: Potential for confusion if resources created via legacy endpoints lack custom ID mappings.

### 8.10 Error Message Inconsistency

**Observation**: Error responses are plain text strings (not JSON). The format varies between resources (e.g., `"Invalid namespace provided: {namespace}"` vs. `"Decorator with ID {id} does not exist in namespace: {namespace}"`). There is no standard error response schema.

**Impact**: API consumers cannot rely on a consistent error structure for programmatic error handling.

### 8.11 ADR Status Transitions

**Observation**: The ADR status update endpoint (`POST /.../{adrId}/status/{status}`) accepts any valid status value without validating the transition. For example, moving from `accepted` back to `draft` or from `rejected` to `accepted` appears to be permitted.

**Impact**: No enforcement of a valid state machine for ADR lifecycle.

## 9. Improvement Opportunities

### 9.1 Pagination and Filtering

**Recommendation**: Add pagination support (`?page=1&size=20` or cursor-based) to all list endpoints. Add filtering by name, description, and creation date for resource discovery.

**Benefit**: Improved performance for large namespaces; better developer experience for discovery.

### 9.2 DELETE Operations with Soft-Delete

**Recommendation**: Implement `DELETE` endpoints for all resource types using a soft-delete pattern (mark as deleted but retain for audit trail). Add a `?includeDeleted=true` query parameter for administrative recovery.

**Benefit**: Enables cleanup workflows and compliance with data retention policies while preserving audit history.

### 9.3 ADR State Machine Enforcement

**Recommendation**: Implement validation of ADR status transitions. For example:

```
draft → proposed → accepted
                 → rejected
accepted → deprecated
         → superseded
```

Reject transitions that violate the state machine (e.g., `accepted` → `draft`).

**Benefit**: Enforces governance discipline on decision records.

### 9.4 Standardized Error Response Schema

**Recommendation**: Adopt a structured JSON error format aligned with RFC 7807 (Problem Details for HTTP APIs):

```json
{
  "type": "urn:calm:error:namespace-not-found",
  "title": "Namespace Not Found",
  "status": 404,
  "detail": "Invalid namespace provided: xyz",
  "instance": "/calm/namespaces/xyz/architectures"
}
```

**Benefit**: Consistent, machine-readable error responses for API consumers.

### 9.5 Complete RBAC Implementation

**Recommendation**: Complete the RBAC implementation to:
1. Support namespace-scoped filtering on `GET /calm/namespaces` (only show namespaces the user has grants for).
2. Document the interaction between OAuth scopes and RBAC grants clearly.
3. Add RBAC grant management UI or CLI tooling.

**Benefit**: Production-ready access control for regulated environments.

### 9.6 Resource Dependency Tracking

**Recommendation**: Add an explicit dependency graph between resources. For example, an architecture may reference patterns and flows. Tracking these relationships would enable impact analysis ("which architectures are affected by changing this pattern?").

**Benefit**: Change impact analysis and governance workflows.

### 9.7 Webhook / Event Notifications

**Recommendation**: Implement a webhook or event notification system that fires when resources are created, updated, or have status changes. This enables CI/CD integration (e.g., trigger validation pipeline when an architecture is published).

**Benefit**: Enables event-driven architecture governance workflows.

### 9.8 Version Comparison and Diff

**Recommendation**: Add an endpoint to compare two versions of the same resource or two revisions of an ADR, returning a structured diff.

**Benefit**: Review architecture evolution without external tooling.

### 9.9 Consistent Domain Model Immutability

**Recommendation**: Align all domain entities to use the immutable Builder pattern (or document the rationale for mutability where it is intentional). This improves thread safety and reduces potential for accidental state mutation.

**Benefit**: Reduced concurrency bugs; clearer developer intent.

### 9.10 Schema-Resource Binding

**Recommendation**: Track which schema version a resource was validated against. Add a `schemaVersion` field to versioned resources and optionally validate resources against specific schemas on creation.

**Benefit**: Schema evolution tracking; automated compliance validation.
